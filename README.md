# tell-me-everything

> 你打开 Claude Code，说"帮我做个记账app"。然后 AI 生成了 800 行代码。你兴冲冲跑起来——崩了。你花了两个小时改 bug，修完一个冒出三个。你开始怀疑是不是自己姿势不对。你不是。你只是掉进了每个 vibe coder 都会掉的坑：**AI 不知道你在想什么。**

---

## 这个坑长什么样

你脑子里有完整的画面——记账要有分类、要有图表、数据存在本地、界面简洁就行。你觉得这是常识。你觉得 AI 当然知道。

AI 不知道。AI 只拿到了 7 个字。它用这 7 个字猜了 100 个决策。猜对是运气，猜错是常态。

更糟的是——项目做完第一版你以为就完了，结果后面加功能、修 bug 的时候，AI 又回到了猜的状态。你明明一开始说好了"别擅自改"、"先问我再动手"——上下文一长，它全忘了。

**这不是你提示词写得不好。是结构问题。** 你需要一个东西在每次 AI 准备动手之前拦住它，把该问的都问出来。不是提醒一次，是每次都提醒。

---

## tell-me-everything 做什么

一个翻译层。架在你和 AI 之间。不替你写代码，替你把没说的话说出来。

```
你："帮我做个记账app"
  ↓
skill 弹选择题：隐私边界？技术栈？架构？编码风格？测试？部署？
  ↓
你选完（不用打字），skill 生成 CLAUDE.md ← AI 每次对话都会读
  ↓
AI 拿到的不再是 7 个字，是你确认过的结构化输入
  ↓
猜的少了，对的多了
```

**首版交付后不沉睡。** 你后面加功能、修 bug、发布开源——每次 AI 准备动手，硬门还在：

| 阶段 | 硬门 |
|------|------|
| 动工前 | 规模评判（就改一行还是整个新模块？）→ 你确认 |
| 写代码中 | AI 在已确认的边界内操作 |
| 写完后 | 自动提示审查 → 你必须写了日志才能切回"讨论模式" |
| 任何时候 | AI 催你"开始写吧"被拒 2 次 → 锁死，不能再催 |

## 触发

- **新项目**：空目录打开 Claude Code → 自动触发
- **手动**：输入 `/init`
- **模糊需求**：你的第一次请求信息严重不足时自动加载
- **发布/开源**：你说"上线""发版本"→ 自动加载发布安全检查

## 怎么设计的

| 机制 | 解决什么 |
|------|---------|
| **15 维度分级访谈** | S 级（安全/隐私/路径）死也要问。A 级（技术栈/架构）追问一次。B 级（风格/日志）问一次跳过 |
| **43 项默认参数** | 编码风格、日志格式、版本控制……没时间细聊就按默认走，想改随时改 |
| **混合状态门** | AI 靠信号词判断你让不让动手——"实现"=动手，"搞吧"=弹框跟你确认，"别动"=不动。不靠猜 |
| **反推动计数器** | AI 催你"可以写了吧？"被拒 2 次 → 锁死讨论模式，直到你亲口说"实现" |
| **决策原子性** | 一个弹框只问一个决策。"做什么"和"谁来做"分两个问题 |
| **日志硬门** | 不写改动日志不许切回 PLAN。AI 可以忘了写——CLAUDE.md 不会忘。下次对话自动检测 |
| **维护/扩展/发布** | 首版不是终点。加功能走规模评判，发版本走安全检查 |
| **隐式信号解读** | 不问你"我做得好吗"。从你的行为中判断方向对不对 |

## 适用

科班生 / 想学 vibe coding 的人。你知道要做什么，只是不知道怎么让 AI 准确理解你的想法——这个 skill 替你翻译。
