# M7 实现后审查与债务管理

## 实现后完整流程

```
代码修改完成
  → ① 写改动日志（如果用户在 I14 选择了"每次改动自动写"）
  → ② 主动提示审查
  → ③ 审查完成/跳过
  → ④ CLAUDE.md 中【当前模式：IMPLEMENT】→【当前模式：PLAN】
```

## 硬门：日志写入

如果用户在 I14-f 选择了硬门模式（默认）：

**日志必须在 IMPLEMENT→PLAN 切换之前写入。** 不写日志，不许切回 PLAN。

```
规则：
  实现完成 → Agent 先写改动日志（logs/agent/YYYY-MM-DD.md）
  → 日志写完了 → 才能 Edit CLAUDE.md 把【IMPLEMENT】改成【PLAN】
  
  如果 Agent "忘了"写日志：
  → CLAUDE.md 里永远是【当前模式：IMPLEMENT】
  → 下次对话开始，Agent 读到 IMPLEMENT → 检查上次改动是否有对应日志
  → 没有 → 补写 → 改回 PLAN

  文件标记不骗人。Agent 可以"忘"，CLAUDE.md 不会忘。
```

**日志格式（JSON）：**

```json
{
  "timestamp": "HH:MM:SS",
  "summary": "改动内容简要描述",
  "files": ["改动的文件路径"],
  "impact": "影响范围（单文件/跨模块/全局）"
}
```

**日志文件：** `logs/agent/YYYY-MM-DD.md`，同一天多次改动追加到同一个文件。

**Git commit message 同步：** commit 时在 body 中引用日志条目，保持双线可追溯。

## 审查触发

```
日志写完后 → Agent 主动报告：
  "本次改动：X 文件 Y 函数，新增 Z 行，影响范围 A/B/C 模块。
   要不要我现在审查一遍？"
  → 用户说"审" → 调用 /review 和 /simplify
  → 用户说"不用" → 跳过
```

**审查工具：** 使用官方 `/review`（检查正确性+安全问题）和 `/simplify`（检查可简化处）。不自己写审计脚本。

## 代码质量检查（用户语言表述）

不使用术语。用用户能理解的问法：

| 检查项 | 问法 |
|--------|------|
| 耦合度 | "改一个地方会不会莫名其妙坏另一个地方？" |
| 复杂度 | "这段代码以后你自己能看懂吗？" |
| 维护性 | "三周后你要加新功能，是改 3 行还是改 300 行？" |
| 扩展性 | "想加新功能时是要重写整个模块还是加个文件就行？" |
| 死代码 | "有没有写了但永远不会跑的代码？" |
| 重复代码 | "有没有两段差不多的代码？" |

## 重构触发条件

任一满足即建议重构：

- 同一模块被改了 5 次以上
- 一个函数超过 N 行（不暴露"圈复杂度"这个词给用户，内部默认阈值 10）
- 一个文件超过 N 行（用户可自定义）
- 用户说"最近怎么老出 bug"
- 每完成一个 Phase 后例行询问

---

## 重构分档

触发了重构建议后：

```
AI 展示影响范围（文件数、行数、涉及模块）
  → AI 给出建议："本次改动范围小，建议走轻量协议。
       如果你觉得不稳也可以走标准协议。"
  → 用户选择：轻量 / 标准
```

| | 轻量重构 | 标准重构 |
|---|---------|---------|
| 适用 | 单文件内、改动 < 50 行 | 跨文件、跨模块、大改动 |
| 步骤 | ①②⑥ | ①②③④⑤⑥ |
| 风险 | 极低，改坏了一眼能看出来 | 高，可能影响到其他模块 |

**用户选档，不靠 AI 判断。**

## 重构安全协议六步骤

### ① 范围声明
明确写死本次重构范围（哪些文件改、哪些文件不碰）。写入 agent 日志。
"本次重构范围：X/Y/Z 文件。明确排除：A/B/C 文件。"

### ② 基准快照
列出当前受影响的功能点清单，用户确认。
"重构前功能点：1. 简历渲染 / 2. 照片处理 / 3. PDF 导出"

### ③ 成功标准
定义"重构成功"的硬性标准（至少满足一条）：
- 总行数减少
- 文件数减少（拆分不算减少）
- import/include 依赖数减少
- 函数参数个数减少（超过 4 个的函数变少）

不满足则不重构。

### ④ 执行
只改范围内文件。发现越界 → 立即停止 → 报告用户：
"重构 A 发现 B 也有问题。不停手会 B→C→D 连锁。先记下 B 的问题，本次只做完 A。"

### ⑤ 回归验证
逐条对②的功能清单。全部通过 → 成功。有失败 → 回滚到重构前。

### ⑥ 终止报告
展示改动对比 + 软硬指标数据：
- 硬性指标自动跑脚本检测（不靠 agent 嘴说）：行数 / 依赖数 / 参数个数 / 测试通过率
- 软性指标展示差异让用户判断：重构前后代码片段 → "你看哪个更容易看懂？"

## 重构效果评判

### 硬性指标（检测脚本，可复现）

| 指标 | 怎么算 |
|------|--------|
| 总行数变化 | 重构后行数 / 重构前行数 |
| 圈复杂度 | 函数是否超阈值（默认 10，不对用户暴露这个词，只说"分支太多"） |
| 重复代码块 | 相同逻辑出现 ≥N 次（N 用户可选，默认 3） |
| 文件依赖数 | import/include 行数变化 |
| 函数参数个数 | 是否有函数超过 4 个参数 |
| 测试通过率 | 100% 通过，不通过直接回滚 |

### 软性指标（用户判）

| 指标 | AI 做什么 |
|------|----------|
| 可读性 | 展示重构前后代码片段："你觉得哪个更容易看懂？" |
| 修改影响面 | 展示受影响文件列表："加同类功能，重构前改 N 个，重构后改 M 个" |
| 职责单一 | "重构前这个函数做了 A/B/C 三件事，重构后拆成了 a() b() c()" |
| 意外风险 | "有以下隐式依赖需要注意" |

### 评分框架

```
硬性 ≥4 项改善 → 硬性通过
硬性 <3 项改善 → 不通过，回滚
硬性通过 + 软性（可读性/修改影响面至少过一项）→ 重构成功
硬性通过 + 软性不过 → 代码瘦了但没变好，回到步骤③重新思考
```

### 核心原则

- **事实可验证，判断归用户。**
- **AI 不下结论，只展示差异。**
- **AI 可以给意见，但意见必须标注为"我的看法"，且永远排在事实后面。**
- **不追求零债务，追求债务可控。**
- **不追求完美架构，每次比之前好一点就算赢。**
