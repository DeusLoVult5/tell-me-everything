# M7 审查卡片

## 审查触发 — 卡片

```
实现完成 → 切回 PLAN → Agent 主动报告：

  "本次改动：{文件列表}，新增/修改 {N} 行，影响 {范围}。
   要审查吗？"

  → 审 → 调用 /review + /simplify
  → 不用 → 跳过（日志仍照写）
```

## 代码质量检查（用户语言，不用术语）

| 检查 | 问法 |
|------|------|
| 耦合 | "改一个地方会不会莫名其妙坏另一个地方？" |
| 复杂度 | "这段代码以后你自己能看懂吗？" |
| 维护性 | "三周后加新功能，改 3 行还是 300 行？" |
| 扩展性 | "加新功能要重写整个模块还是加个文件？" |
| 死代码 | "有没有写了但永远不会跑的代码？" |
| 重复 | "有没有两段差不多的代码？" |

## 重构触发 — 卡片

```
任一满足即建议重构：
  □ 同一模块改了 ≥5 次
  □ 函数/文件超过阈值
  □ 用户说"最近怎么老出bug"
  □ 每完成一个 Phase
```

## 重构分档 — 卡片

```
触发后：
  Agent 展示影响范围 → 建议轻量/标准 → 用户选档

  轻量（单文件 <50行）→ ①范围声明 ②基准快照 ⑥终止报告
  标准（跨文件/大改动）→ ①②③④⑤⑥ 全流程
```

## 重构六步骤 — 卡片

```
① 范围声明    写死改哪些文件、不碰哪些
② 基准快照    列出受影响的功能点 → 用户确认
③ 成功标准    至少满足一条：行数↓ / 依赖↓ / 参数↓
④ 执行        只改范围内，越界即停 → 报告用户
⑤ 回归验证    逐条对②的功能清单 → 全通过=成功 / 有失败=回滚
⑥ 终止报告    展示改动对比 → "你看哪个更好？"
```

## 效果评判 — 卡片

```
硬性（可复现检测，不靠嘴说）：
  行数↓ / 圈复杂度 / 重复代码 / 依赖数 / 参数数 / 测试通过率
  ≥4项改善 → 硬性通过。≤2项 → 不通过=回滚。

软性（用户判）：
  Agent 展示重构前后差异 → "你看哪个更好？"
  AI 不下结论，只展示差异。事实排前，"我的看法"排后。

硬性通过 + 软性（可读性/影响面至少过一项）→ 重构成功。
硬性不过 → 回滚。
```
