# 后段协议：维护、扩展与发布

首版交付后，硬门不关。每一轮改动都走同一个最小门控序列。

---

## 核心原则

- **AI 不替用户决定"要不要做"和"做出来对不对"，但"怎么做"是 AI 的。**
- **门不是用完即弃，是每次必过。**
- **规模由 AI 陈述事实 + 用户确认，不由 AI 默默判定。**

---

## 任何改动请求的完整流程

```
用户提出改动
  → 步骤 0：M0 四要素追问（如果用户描述不完整）
     在哪/做了什么/什么现象/期望什么
     不问完不动手

  → 步骤 1：规模评判
     Agent 陈述事实 + 建议规模，弹 AskUserQuestion 确认

  → 步骤 2：按规模加载最低问询集（见下表）

  → 步骤 3：写入 CLAUDE.md：
     【当前模式：IMPLEMENT】
     【规模：X改动 — N文件】
     如果 CLAUDE.md 里没有规模声明但模式是 IMPLEMENT →
     状态矛盾 → 说明 Agent 跳过了步骤1 → 下次对话会被检测到

  → 步骤 4：实现
     在已确认的边界内写代码

  → 步骤 5：M7 审查 + 日志硬门
     日志必须写，不写不许切回 PLAN

  → 步骤 6：清除规模声明 + 切回 PLAN
     CLAUDE.md：【当前模式：PLAN】
```

---

## 规模分级

| 级别 | 条件 | 最低问询集 | 典型场景 |
|------|------|-----------|---------|
| **微量** | 单个已存在文件、不创建新文件、不引入新依赖 | M5 门 + M7 审查 + 日志硬门 | 改一行文案、修一个参数值、改一个 CSS 颜色 |
| **小改动** | 1-3 文件、可能创建新文件、无新依赖 | 微量 + P4（怎么测）+ M4（有参考吗） | 加一个按钮、修一个 bug、改一个函数的逻辑 |
| **中改动** | 跨模块、或需要新依赖、或改数据流 | 小改动 + 追加 A 级：I5(技术栈要扩？) I6(架构要调？) I12(新三方库？) | 加导出功能、换数据库驱动、加一个新的 API 端点 |
| **大改动** | 新子系统/新模块/架构变更 | 完整 S+A+B。已有答案从 CLAUDE.md 继承，**只确认变了什么** | 加支付模块、前后端拆分、上新框架 |

---

## 修改类型分流

任何改动请求在规模评判之后，按修改类型分两条路径：

```
判断类型：
  ├─ 代码类（源码/配置/脚本）→ 走完整门控流程（M5→规模→实现→M7→日志→切PLAN）
  └─ 文档类（README/CHANGELOG/说明/SKILL.md/references）→ 走文档修改协议
```

### 文档修改协议

**Agent 不直接写入源文件。任何文档修改，一律走副本流程。** 不存在"改一行不用走副本"的例外。

```
① 复制原文件 → 文件名加后缀 _draft（如 README_draft.md）
② 在草案副本上修改
③ 告知用户：
   "原文件：[路径]
   草案副本：[路径_draft.md]
   打开两个文件对比看效果。"

④ 弹 AskUserQuestion：
   - 合并（用草案替换原文件，删除草案）→ Agent 执行替换 + Git commit
   - 保留两个（都不删，用户自己后续处理）→ 原文件不动，不产生额外 commit
   - 丢弃草案（删草案，原文件不动）→ 无额外操作

⑤ 日志照写，走日志硬门
```

**为什么直接改源文件就是偷懒：** 直接改源文件等价于 Agent 替用户做了"合并"这个决定。回滚是用户的事——Agent 只负责输出，不负责"不满意怎么办"的兜底。

## 规模声明的格式（写入 CLAUDE.md）

```
【当前模式：IMPLEMENT】
【规模：小改动】
【预计改动：app.py L30-L80 / template.html L15-L20 / test_app.py L10-L40】
【新依赖：无】
【验证方式：python -m pytest test_app.py】
```

---

## 大小改动判断的规则（Agent 自查用，不替用户判）

| Agent 可自查的事实 | 如何获取 |
|-------------------|---------|
| 涉及几个文件 | git diff --stat 或文件内容理解 |
| 会不会跨模块 | 看 import 链 |
| 会不会新增依赖 | 对比 requirements.txt 的 diff |
| 会不会改数据流 | 函数签名是否变化 |

用户确认的是**级别**和**方向**，不是这些事实。

## 发布节点

当用户说以下任一关键词时，加载发布子集：

**触发词：** "上线" / "发布" / "开源" / "发版本" / "release" / "部署"

### 发布子集问询

| # | 问什么 | 为什么 |
|---|--------|--------|
| R1 | 版本号怎么定？语义化版本（x.y.z）？ | 发布前必须定，否则不可追溯 |
| R2 | CHANGELOG 更新吗？ | 用户知道你改了，用户不知道你改了什么 |
| R3 | 代码里有没有不想公开的敏感信息（API key、密码、内部路径）？ | 开源前最后一关 |
| R4 | README 全了吗？安装/使用/配置写清楚了吗？ | 别人拿到能跑 |
| R5 | 选什么开源协议？默认 MIT（最宽松） | 以防万一 |
| R6 | 发行方式：源码（GitHub Release）/ PyPI 包 / Docker 镜像？ | 不同项目不同方式 |

### 出版本流程

```
发布节点触发
  → R1-R6 逐一确认
  → 更新 CHANGELOG
  → git tag vX.Y.Z
  → 按用户选择的发行方式执行
```

---

## 常见防偷懒检查点

| Agent 可能偷懒的路径 | 硬门怎么拦 |
|---------------------|-----------|
| 跳过规模声明直接实现 | CLAUDE.md 里 PLAN 没变成 IMPLEMENT → 下次对话被检测 |
| 跳过 M7 审查 | 审查是日志硬门的前置 → 不审就没日志，没日志就切不回 PLAN |
| 跳过日志 | CLAUDE.md 永远是 IMPLEMENT → 下次对话 Agent 发现状态矛盾 |
| 微量级滥用到什么都是微量 | 用户确认规模时看到"微量"但实际改了 5 个文件 → 用户驳回 |
| 发布时不做安全检查 | R3 必须在发布流程中，不能跳过 |
