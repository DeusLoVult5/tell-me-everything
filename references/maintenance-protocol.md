# 后段协议卡片

**首版交付后，硬门不关。每次改动走同一个最小门控序列。**

## 规模分级 — 卡片

| 级别 | 条件 | 最低问询 | 典型场景 |
|------|------|---------|---------|
| **微量** | 单文件、不新建、无新依赖 | M5门+M7+日志 | 改一行文案/修参数值 |
| **小改动** | 1-3文件、可新建、无新依赖 | 微量+P4+M4 | 加按钮/修bug/改函数逻辑 |
| **中改动** | 跨模块/新依赖/改数据流 | 小改动+I5+I6+I12 | 加导出/换驱动/加API |
| **大改动** | 新子系统/新模块/架构变更 | S+A+B，只确认变了什么 | 加支付/前后端拆分 |

## 修改类型分流 — 卡片

```
代码类（源码/配置/脚本）→ 完整门控（M5→规模→实现→M7→日志→切PLAN）
  → 方案必须在对话呈现。不以文件代替。

文档类（README/CHANGELOG/skill文件）→ 副本流程（见下）

同步更新：CHANGELOG/README附属更新，走副本但不弹合并框
  → 三个判定条件缺一不可（见下）
```

## 文档修改协议 — 卡片

```
① cp → _draft.md
② 改草案
③ 告知路径："原文件+草案路径，打开对比"
④ 弹 AskUserQuestion：合并 / 保留两个 / 丢弃草案
⑤ 日志照写
```

## 同步更新豁免 — 卡片

三个判定条件（必须同时满足）：

```
✓ 前面紧邻完整 IMPLEMENT 链路（规模声明→实现→M7→日志）
✓ 内容仅限于记录那次改动的 CHANGELOG 条目或 README 一行
✓ IMPLEMENT 尚未切回 PLAN（在收尾环节）

缺任一 → 不是同步更新 → 走完整确认流程
```

## Pre-Implementation 自检清单 — 卡片

步骤 3.5 — 动手前逐条确认。**不是看一眼，是逐条确认。**

```
☐ 1. 规模声明已在对话呈现并由用户确认？
☐ 2. 输出完整呈现？（不以"其余不变"代替，不以文件代替对话）
☐ 3. S 级信息全部确认？（环境/路径/文件来源不能代定）
☐ 4. 有推结束的暗示？（"可以写了吧/同步push"→ 停止）
☐ 5. 文档走副本流程？(_draft → 对比 → 合并/丢弃)
☐ 6. 以"同步更新"为名绕过确认？（判定三条件 ✓✓✓？）
☐ 7. 有 Bash 绕过钩子？（sed -i / cat > / tee / echo >> → 走门控）
☐ 8. "[动词]完告诉我效果"被误判为实现信号？（谓语"告诉/给/看"→ 不实现）
```

**全部 ✓ → 进入实现。任一 ✗ → 停下。**

## 发布节点 — 卡片

触发词："上线"/"发布"/"开源"/"发版本"/"release"/"部署"

```
→ R1 版本号怎么定？（语义化版本x.y.z）
→ R2 CHANGELOG 更新？
→ R3 敏感信息检查（API key/密码/内部路径）
→ R4 README 齐全？
→ R5 开源协议？默认 MIT
→ R6 发行方式？源码/PyPI/Docker
→ 更新 CHANGELOG → git tag vX.Y.Z → 执行
```

## 防偷懒检查点 — 卡片

| 偷懒路径 | 硬门 |
|---------|------|
| 跳过规模声明直接实现 | PreToolUse hook: 无`【规模：】`→deny Write/Edit |
| 改模式绕过声明（PLAN→IMPLEMENT） | 规模声明是唯一通行证，每次留审计痕迹 |
| 跳过 M7 | 审查是日志前置 → 不审就没日志 |
| 跳过日志 | Stop hook + IMPLEMENT 矛盾检测 |
| 微量级滥用 | 用户确认时看到"微量"但改了5文件→驳回 |
| 跳过发布安全检查 | R3 硬性包含在发布流程中 |
| 写到项目目录外 | check-boundary.py → deny |
