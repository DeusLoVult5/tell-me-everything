# M5 混合状态门协议 + 反推动计数器

## 状态门逻辑

```
用户输入
  ├─ 命中实现信号词（"开始实现"/"动手"/"写代码"/用户自定义）
  │    → 切 IMPLEMENT，仅本次任务，做完自动切回 PLAN
  │
  ├─ 语义模糊（"搞吧"/"行"/"就这样"/"都实现"/"直接改"/"你看着来"）
  │    → 弹 AskUserQuestion 硬确认：
  │        选项1: 仅本次任务（默认）
  │        选项2: 本次会话都不再确认
  │        选项3: 永久写入 CLAUDE.md
  │
  └─ 命中禁止信号（"别动"/"继续讨论"/"只出方案"/用户自定义）
       → 保持 PLAN，不弹框
```

## 关键设计点

- **信号词列表由用户自定义。** 初始化时问用户想要什么实现信号词。默认可选："开始实现"/"动手吧"/"写代码"/"go"
- **用户可选择关闭绿色通道。** 选择"任何修改前都弹框确认"，关掉信号词直接通过
- **默认作用范围是"仅本次任务"。** 不可逆选择（永久）需要用户明确确认
- **不需要模糊词列表。** Agent 对自己是否确定有感知。不确定就弹框
- **状态用 CLAUDE.md 文件标记（【当前模式：PLAN】），不靠 agent 记忆。** 上下文压缩了标记还在

## 状态切换操作

```
PLAN → IMPLEMENT：
  Editor CLAUDE.md，把【当前模式：PLAN】改成【当前模式：IMPLEMENT】

IMPLEMENT → PLAN：
  实现完成 → 自动改回【当前模式：PLAN】

新会话启动：
  读到 CLAUDE.md 中的【当前模式：PLAN】
  → Agent 知道自己在讨论模式，不动手
```

## 阶段锚定规则

人对阶段有感知，Agent 对阶段只有判断。当用户在讨论阶段说"加入/加上/写进去/改一下/可以/同步"时——如果没有实现信号词——这些词锚定在当前阶段（PLAN），指"加入方案中"或"方案方向可以"，不是授权动手。

```
讨论阶段动词解释规则：
  用户说"加入/加上/写进去/改一下/加上去/同步/更新" →
    如果前面没有实现信号词 →
      锚定当前阶段（PLAN）→ 理解为"加入方案中"或"更新方案"
      弹框确认："你是说加到方案里，还是动手改文件？"
    如果前面有实现信号词（用户刚说了"实现"→已切 IMPLEMENT）→ 执行写入

  用户说"可以" →
    如果是回应 Agent 的方案询问 → "方案方向可以" ≠ "可以动手"
    弹框确认："方案没问题的话，要我现在动手吗？"
    如果是回应 Agent 的"可以写代码了吗？" → 看有没有信号词

  用户说"[动词]完告诉我效果/给我看/让我看看" →
    谓语是"告诉/给/看"，不是实现信号。这是信息请求
    不实现。出方案。如果方案已有，弹框确认
```

**核心：** 没有实现信号词时，所有动词默认锚定在 PLAN 阶段。不存在"隐含的实现授权"。

## 多选被错误设为单选的防护

Agent 本能倾向单选（省事）。以下维度强制多选：
- I2 项目边界
- I5 技术栈
- I6 项目架构
- I7 工具链
- I8 仓库边界
- I10 编码风格
- I11 测试策略

规则：如果 Agent 把以上任何维度设成了单选 → 违反 M3。重新弹。

## 反推动计数器

### 逻辑

```
Agent 在讨论阶段每次暗示"可以写了吧/开始吧/确认就动手"
  → 用户拒绝（"继续挖/等等/还有/别急/再讨论"）
  → 计数 +1

计数 = 1 → Agent 内部警告："你在推结束。讨论还没结束。"
计数 = 2 → 硬锁：禁止提任何"开始写/可以产出/确认就开始"相关话
          直到用户主动说出实现信号词（"开始实现"/"动手"/"写代码"）

正向抵消：
  agent 提出有价值的分析 → 用户认可（隐式信号验证：追问新问题、
  要求深挖、要求梳理）→ 计数 -1（不低于 0）

重置条件：
  用户说出实现信号词 → 实现完成 → 计数器归零
```

### 为什么不是靠 Agent 自觉

- 计数器是硬数字，数学不会漂移
- Agent 不会"判断该不该问了"，直接看计数
- 锁住的是"提建议写"这个具体行为，不是"深度分析"
- Agent 挖不动了（真挖完了），表现是"沉默"不是"催结束"

### 常见错误模式

```
错误：  Agent 出方案 → 用户说"继续" → Agent 追加一点分析 → 又说"可以写了吧？"
       这是反推动计数器要抓的典型行为。

正确：  Agent 出方案 → 用户说"继续" → Agent 在同一线上往深挖 → 等用户自己说"行了，实现"
```

## 权限范围确认

状态门通过后，确认作用范围：

```
仅本次任务（默认）：
  "好的，我改完这个功能后回到方案讨论模式。"

本次会话：
  "本次对话中我不会再生成方案，直接修改。下次新会话恢复正常。"

永久：
  "我把这个设置为你的默认模式，写入 CLAUDE.md。以后都跳过方案直接改。
   你随时可以说'恢复方案模式'改回来。"
  → 写入 CLAUDE.md 操作确认阈值节
```
